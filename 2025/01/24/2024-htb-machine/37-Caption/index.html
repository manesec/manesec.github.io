在patch中打了兩個非預期，本篇文章前兩個都會介紹打的内容，另外預期在本篇文章的中後面部分。

從更新的日志中可以看到，這台機器打了一些補丁，尤其是修復了兩個非預期漏洞。首先，通過 8080 端口訪問代碼倉庫時，發現可以使用默認的 root 賬號和密碼登錄，這揭示了系統的安全漏洞。進入倉庫後，發現有關於網站的代碼和提交日志，並且注意到 80 端口的 Haproxy 配置禁用了某些路徑。另一個程序監聽 9090 端口，代碼中存在命令注入漏洞。進一步分析顯示，系統使用 H2 數據庫，並且在 margo 用戶的目錄下。利用這些信息，可以通過特定的漏洞獲得遠程代碼執行權限。接著，通過 WhatWeb 工具，發現該網站使用 Python 框架，並且有 Varnish 緩存服務器。這提示可以利用緩存投毒技術，並嘗試通過 XSS 攻擊來繞過 Haproxy 限制。通過多次嘗試，最終成功獲得了用戶 shell。進一步分析發現，通過利用 CopyParty 框架的漏洞，可以讀取任意文件，進而獲得 SSH 密鑰。最終，通過分析 RPC 服務器的源碼，利用正則表達式的漏洞，成功提權獲得 root 權限。

在預期的路徑中，使用了 h2csmuggler 工具來進行請求走私，成功繞過 Haproxy 的限制，並利用緩存投毒技術來執行 XSS 攻擊。通過不斷嘗試和調整，最終成功獲得了管理員的 cookie，進一步訪問受限的日誌文件。利用 CopyParty 框架的漏洞，成功讀取了 SSH 密鑰，並使用這些信息獲得了用戶權限。通過分析 RPC 服務的 Thrift 文件，設計了一個客戶端來調用服務，最終利用命令注入漏洞成功提權至 root。這一系列操作展示了如何從初始訪問到最終提權，利用系統中的多個漏洞來達成目標。

也就是你现在看的这个页面其实是26秒之前的页面，而不是实时的页面，

缓存服务器的原理就是第一次请求会被记录到数据库， 之后用户访问就是从缓存好的数据里面提取，但是会有一个超时的问题也就是说开发者会指定多少秒之后强制重新刷新缓存，確保内容會同步，這個時間也就是所谓的緩存内容过期時間。

在 log_service.thrift​ 文件中，您可能會找到以下內容：

  1. 服務定義：定義了可用的服務方法，例如記錄日誌、檢索日誌等。
  2. 數據結構：定義了用於服務的數據類型，例如日誌條目、錯誤信息等。
  3. 傳輸和協議設置：指定如何在客戶端和服務器之間傳輸數據。

這個文件通常是用來生成相應語言的代碼，以便開發者可以在不同的應用程序中使用這些服務。

Analysis source code for Logservice

然後翻一下 server.go​ 的源碼，所以去看一下RPC服務器的源碼，看一下会做些什么事情。

你可以看到他是打開一個文件，然後進行正則匹配：

  1. IP 地址匹配 (​​ipRegex​​ ) :

    • 這個正則表達式用於匹配 IPv4 地址。IPv4 地址由四個數字組成，每個數字的範圍是 0 到 255，並且用點（.​）分隔。
    • ​\b​ 是單詞邊界，確保匹配的 IP 地址不會是其他數字的一部分。
    • ​(?:\d{1,3}\.){3}​ 表示匹配三個由一到三位數字組成的部分，後面跟著一個點（.​）。
    • ​\d{1,3}​ 最後一部分匹配最後一個數字，這樣整個表達式就能匹配完整的 IP 地址格式。

  2. 用戶代理匹配 (​​userAgentRegex​​ ) :

    • 這個正則表達式用於提取用戶代理字符串。用戶代理通常是在 HTTP 請求中發送的，包含了有關客戶端應用程序的信息。
    • ​"​ 符號用於匹配字符串的開始和結束。
    • ​user-agent​ 是要匹配的鍵名。
    • ​:​ 是鍵值對中的分隔符。
    • ​([^"]+)​ 是一個捕獲組，匹配用戶代理的具體內容，直到遇到下一個引號（"​）。這部分會捕獲用戶代理的實際字符串。

最後他是通過正則提取后塞進去 echo 'IP Address: %s, User-Agent: %s, Timestamp: %s' >> output.log​ 裏面。

其中第一個ip的格式是固定的，第二個可以自由隨意更改，

更關心的是第二個地方這個： "user-agent":"([^"]+)"​

​​
